Project Overview

This is a radio streaming application combining a PHP/MySQL backend with a JavaScript/HTML5 frontend.

  • Stream MP3 files from local server
  • Upload music with metadata
  • Manage dynamic playlists

Audio Features

  • Real-time audio visualization
  • Advanced playback controls
  • Shuffle and repeat modes

Database Configuration

Database Creation 

-- Create database
CREATE DATABASE loco_music;
USE loco_music;

Songs Table Schema 

CREATE TABLE songs (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  file VARCHAR(255) NOT NULL,
  cover VARCHAR(255),
  artist VARCHAR(255) NOT NULL,
  lyrics TEXT,
  uploaded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

This schema supports all core features including metadata storage for songs, cover art references, and lyrics. The uploaded_at timestamp enables chronological playlist sorting.

PHP Configuration

Database Credentials Setup

Database credentials are now managed via a `db_config.php` file, separate from the main `index.php` script.

  1. Locate the `db_config.sample.php` file in the project root.
  2. Create a copy of this file and name it `db_config.php`.
    cp db_config.sample.php db_config.php
  3. Open `db_config.php` and enter your actual database connection details (host, username, password, database name).
// Example content of db_config.php:
define('DB_HOST', 'your_localhost');
define('DB_USERNAME', 'your_db_user');
define('DB_PASSWORD', 'your_db_password');
define('DB_NAME', 'loco_music');

Important Security Note

The `db_config.php` file is included in the project's `.gitignore` file. This is intentional to prevent your sensitive database credentials from being accidentally committed to a version control repository like Git. Keep your `db_config.php` file secure and do not share it publicly.

.htaccess File Explained

1

URL Rewriting

<IfModule mod_rewrite.c>
    RewriteEngine On
    RewriteBase /
    RewriteCond %{REQUEST_FILENAME} !-f
    RewriteCond %{REQUEST_FILENAME} !-d
    RewriteRule ^(.*)$ index.php [QSA,L]
</IfModule>

Enables clean URLs by routing all requests to index.php unless the request is for an existing file or directory.

2

File Access Security

# Disable directory browsing
Options -Indexes

# Disallow remote access to sensitive files (including db_config.php)
<FilesMatch "\.(env|ini|log|sql|bak|sh|config\.php)$">
    Order Allow,Deny
    Deny from all
</FilesMatch>

Prevents directory listing and blocks access to configuration/backup files, now explicitly including files like `db_config.php` through the `config\.php` pattern.

3

PHP Settings

# Note: These settings can often be configured via your hosting control panel,
# which may override .htaccess directives.
<IfModule mod_php.c> # Uses mod_php.c for broader compatibility
    php_value upload_max_filesize 64M
    php_value post_max_size 64M
    php_value max_execution_time 300
    php_value max_input_time 300
</IfModule>

Configures PHP for file uploads. The directive `mod_php.c` is used for better compatibility across different PHP versions managed by Apache.

4

HTTP Security Headers

Several HTTP security headers have been added to enhance protection against common web vulnerabilities. These are configured within an <IfModule mod_headers.c> block:

  • X-Content-Type-Options "nosniff": Prevents the browser from trying to guess the content type of a file, reducing risk of XSS attacks.
  • X-Frame-Options "SAMEORIGIN": Protects against clickjacking by preventing the page from being embedded in iframes from different origins.
  • Referrer-Policy "strict-origin-when-cross-origin": Controls how much referrer information is sent with requests, enhancing user privacy.
  • Permissions-Policy: Disables potentially sensitive browser features like geolocation, microphone, camera access by default.
  • Content-Security-Policy (CSP): A basic CSP is implemented to define allowed sources for scripts, styles, fonts, images, and media, mitigating XSS and data injection attacks. The current policy allows inline scripts/styles for compatibility but can be strengthened by moving these to external files.

Example (snippet from the CSP): Header set Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.tailwindcss.com https://cdnjs.cloudflare.com; ..."

5

MIME Types

<IfModule mod_mime.c>
    AddType audio/mpeg .mp3
    AddType image/jpeg .jpg .jpeg
    AddType image/png .png
    AddType image/gif .gif
</IfModule>

Ensures correct content-type headers are sent for media files.

6

Performance Optimization

<IfModule mod_deflate.c>
    AddOutputFilterByType DEFLATE text/html text/plain text/xml text/css application/x-javascript application/javascript application/json
</IfModule>

<IfModule mod_expires.c>
    ExpiresActive On
    ExpiresByType audio/mpeg "access plus 1 week"
    ExpiresByType image/jpg "access plus 1 week"
    ExpiresByType image/jpeg "access plus 1 week"
    ExpiresByType image/png "access plus 1 week"
    ExpiresByType image/gif "access plus 1 week"
    ExpiresByType text/css "access plus 1 day"
    ExpiresByType application/javascript "access plus 1 day"
</IfModule>

Enables GZIP compression and sets caching headers for static assets to improve performance.

Core Features

Music Streaming System

  • Progressive loading with time/duration display

Playlist Management

Upload System

Audio Visualization

Security Considerations

Summary of Recent Security Enhancements:

  • Database Credentials: Moved from `index.php` to `db_config.php` (which is gitignored).
  • File Uploads: Added server-side MIME type validation for songs and cover art. Implemented filename sanitization.
  • CSRF Protection: Added CSRF token validation to the song upload form.
  • XSS Prevention (Lyrics): Lyrics are now sanitized using `htmlspecialchars` before database storage.
  • HTTP Security Headers: `.htaccess` now includes `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`, and a basic `Content-Security-Policy`.

Database Credential Management

Status: Improved. Credentials are in `db_config.php` (gitignored).
Recommendation: Ensure `db_config.php` has strict file permissions on the server and is not web-accessible.

File Upload Security

Status: Improved. Server-side MIME type validation and filename sanitization are implemented.
Note: Robust file upload security is an ongoing concern. Evaluate risks if used in sensitive environments.

SQL Injection Risk

Status: Partially Mitigated. Prepared statements are used for song uploads.
Recommendation: Review all database interactions to ensure consistent use of prepared statements or proper escaping.

CSRF Vulnerability

Status: Improved. The song upload form is protected by CSRF tokens.

XSS Vulnerability (Lyrics)

Status: Improved. Lyrics are sanitized using `htmlspecialchars` before storage.
Recommendation: Always ensure proper output encoding on the client-side as a defense-in-depth measure.

HTTP Security Headers

Status: Implemented. `.htaccess` configures several security headers.
Recommendation: Refine the Content-Security-Policy by eliminating 'unsafe-inline' for scripts/styles for higher security.

General Security Advice

  • Keep server software (Apache, PHP, MySQL) and any libraries up to date.
  • Implement robust input validation for all user-supplied data beyond what's currently done.
  • Regularly back up your database and application files.
  • Consider using HTTPS for encrypted communication.
  • If handling many users or sensitive data, conduct professional security audits.
